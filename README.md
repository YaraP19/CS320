# CS320
Software Test, Automation QA 2026

Throughout the course, I created the Appointment, Contact, and Task features, which required me to think like both an engineer and a quality checker. I did not treat testing as a secondary task. Instead, I believed that every line of code had to show it could handle real situations, special cases, and unexpected issues. This meant I had to design validation rules carefully and test them thoroughly using JUnit. I questioned the code instead of just trusting it. This approach helped me build a system that worked well and was strong.

My design process begins with understanding what users expect and putting those expectations into clear rules in the code. I identify key requirements, such as which fields are mandatory, their length limits, and the valid date ranges. Then, I create model classes to validate these rules and service classes to manage the objects effectively.

I use a straightforward, test-driven method. First, I implement the basic logic. Then, I write unit tests to check it and refine the code based on the test results. I have faced challenges, especially when I lost focus on specific requirements. Small mistakes led to validation errors that the tests quickly identified. These experiences taught me the importance of slowing down, reviewing the requirements, and using the tests to guide corrections. Being organized helps ensure that the final software is correct, easy to update, and meets users' needs.
